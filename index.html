Abstract

This project developed and validated a software-hardware integration framework for the Galassia-5 satellite payload subsystem. The main objective was to connect camera payloads, onboard processing hardware, SpaceWire communication, and AI-based image inference into one testable end-to-end pipeline. The work focused on the ZSOM-F01 payload computer with Hailo-8L support, together with camera integration through Ethernet and SpaceWire, Ku-band radio integration through Python-based control, and timing validation of the full image-processing workflow.

The pipeline processes images captured by the camera, stores them in a directory, splits them into 48 tiles to match the 800×800 input requirement of the Hailo HEF model, runs YOLOv8-based inference, converts detections to coordinate output, and generates CSV results. Benchmark testing showed that the system completes the pipeline in approximately 4.3 to 4.5 seconds per image, which is comfortably below the mission requirement of 20 seconds per image. fileciteturn0file0

The project also addressed practical board bring-up issues. The ZSOM-F01 board originally ran PetaLinux, but limitations in internet access, package management, and day-to-day development made an Ubuntu-based root filesystem more suitable during development. The work therefore included boot configuration updates, SD card partitioning, filesystem replacement, and hardware/software environment adjustments.

Introduction

Galassia-5 is a satellite project made up of multiple independently developed subsystems, including camera payloads, onboard processing, communication interfaces, and radio hardware. The main engineering challenge was to integrate these parts into a single software-hardware pipeline that could be tested, verified, and prepared for mission use.

This project focused on the payload side of the system. The payload computer had to communicate with camera hardware, run inference using Hailo-8L, support SpaceWire-based communication for a camera that only supports SpaceWire, and integrate Ku-band radio control through Python. The project also had to meet a strict mission timing requirement: the pipeline for each image could not exceed 20 seconds.

A further challenge was that the final flight camera was not fully fixed at the time of development. Because of this, the project needed to support more than one camera path and prove that both could be integrated into the same overall framework.

Problem Statement

Engineering problem

Subsystems were developed separately, creating a risk of interface mismatch, delayed debugging, and long final integration cycles. The project therefore needed a common integration layer that could connect camera input, board communication, and inference output in a controlled way.

Mission constraint

The image-processing pipeline must complete in less than 20 seconds per image. The tested pipeline achieved approximately 4.3–4.5 seconds per image, leaving a strong performance margin for mission use. fileciteturn0file0

In practical terms, the problem was not just to make the camera communicate or to make the AI model run. The real task was to connect all of the relevant subsystems so the payload path could be demonstrated in a realistic and mission-relevant environment. This included board bring-up, SDK setup, hardware networking, camera access from Python, and end-to-end timing measurement.

This timing margin is one of the strongest outcomes in the project because it proves the pipeline is not just functional, but also mission-acceptable.

System Architecture

Pipeline flow

Camera image capture → image saved to directory → image tiled into 48 sections → Hailo YOLOv8 inference → detections converted to coordinates → CSV output generated.

The 48-tile split exists because the HEF model expects an 800×800 input size.

The output CSV provides a compact format for downstream use and makes it easier to compare detections across tests.

Hardware role

The ZSOM-F01 payload computer acts as the central integration platform. It is the mission-representative hardware, so it is the most valid target for final verification. The board ties together camera control, Python software execution, Hailo acceleration, and communication with other payload subsystems.

Integration Work Completed

Camera payload integration

Integrated the camera input path with the payload computer and the Hailo-8L processing chain. For the Skunkworks camera, the Ethernet connection required careful network setup because the board has two Ethernet ports and only eth0 could be used reliably. DHCP was configured so that the camera and the ZSOM-F01 board shared the same network and could communicate correctly.

SDK and Python access

Installed and configured the Lucid Atlas camera SDK, downloaded the required drivers, and set up Python access so the camera could be controlled from the payload computer using the SDK interface, including arena_api.system import system. This step was essential because it made the camera accessible from the same software environment used for the rest of the pipeline.

SpaceWire node integration

Built and tested the SpaceWire path to support the Siemera camera, which only supports SpaceWire. Although only one camera may ultimately be flown, supporting both paths reduced mission risk and gave the project a more flexible integration architecture.

AI inference pipeline

Implemented image tiling, YOLOv8 inference, duplicate-detection handling, and CSV output generation. The pipeline also supports timing comparison between single-image and batch-style processing, which was used to identify the dominant runtime costs.

Ku-band radio integration

Integrated the Ku-band radio into the pipeline through Python-based control calls, extending the same scripting-based workflow used for the camera subsystem. This showed that the software control approach could scale across multiple mission subsystems.

Software Environment and Board Bring-Up

The ZSOM-F01 board was originally based on a PetaLinux image, but this created practical limitations during development, especially around internet access, package management, and command-line convenience. To make development easier, the board was migrated to an Ubuntu-based root filesystem for testing and integration work.

The bring-up process required editing the boot and root partitions on the SD card. After applying the patch and fixing the system-user.dtsi file, the boot image was rebuilt and the required boot files were copied into the boot partition. The root partition was then replaced with the Ubuntu filesystem, while keeping the required modules and firmware directories so the system could still boot correctly.

Because the files in the SD card partitions were protected, root-level access was needed to copy the Ubuntu filesystem into place. A boot configuration file also had to be updated so that U-Boot would mount and start the Ubuntu filesystem instead of the default PetaLinux root filesystem. In effect, the boot chain became: bootloader → U-Boot → kernel → Ubuntu filesystem.

This change made software development easier, but it also revealed a trade-off: Ubuntu was larger and ran hotter on the ZSOM-F01 board, so the project may need to return to PetaLinux for the final mission configuration. The current hardware notes also indicate that the USB on the existing PCB board is damaged, which prevents normal GUI booting and motivates a future redraw of the top.xsa hardware configuration for a replacement board.

Hailo vs CPU, and ZSOM-F01 vs Raspberry Pi

A key design decision in this project was whether image inference should run on the CPU or on the Hailo-8L accelerator. The CPU-only approach was tested as a baseline, but the measured runtime was not suitable for mission use. On the Raspberry Pi test platform, CPU-based batch processing exceeded the 20-second per image limit at approximately 22 seconds per image, while single-image processing reached approximately 42 seconds per image. The CPU pipeline also terminated after several images because of insufficient memory. These results show that CPU-only inference is not viable for the Galassia-5 payload pipeline.

By contrast, Hailo-based inference provided a much better performance profile. On the ZSOM-F01 payload computer, the full pipeline completed in about 4.3 to 4.5 seconds per image, which is well below the 20-second mission requirement. The measured timing breakdown showed that tiling took approximately 2.25 seconds and inference took approximately 2.08 seconds, with only a small one-time model initialisation cost in warm runs. fileciteturn0file0

Why Hailo

Hailo was selected because it provides fast, lightweight inference that fits the satellite payload constraint much better than CPU-only execution. It reduces inference time, keeps the pipeline within the timing budget, and avoids the resource pressure that appears when the same workload is executed entirely on a general-purpose CPU.

Why ZSOM-F01

ZSOM-F01 was chosen because it is the mission-representative payload computer for Galassia-5. Raspberry Pi is useful as a development baseline, but it is not the final spacecraft computer. Therefore, the ZSOM-F01 plus Hailo combination is the correct final integration target.

The correct conclusion is therefore not simply that Hailo is faster. Limitations

The project did have limitations, and these should be stated clearly in the report:

  • the final flight camera choice was not fully fixed;
  • not every subsystem may have reached final flight-grade integration;
  • Ubuntu was useful for development but is heavier and hotter than PetaLinux;
  • the current PCB has damaged USB hardware, limiting normal GUI boot use;
  • more timing comparisons could be useful if another board such as Raspberry Pi 5 is available;
  • the Ku-band subsystem may still need additional finalisation depending on mission scope.

These limitations do not weaken the project. They make the report more credible because they show what still needs to be done and where the engineering trade-offs lie.

Future Work

  • finalise the flight camera selection;
  • complete any remaining Ku-band radio integration;
  • validate the selected configuration on the final mission hardware;
  • test timing on a Raspberry Pi 5 or development PC only if a comparison is useful;
  • return to PetaLinux for the final image if system size or heat becomes an issue;
  • refine the top.xsa hardware configuration for future board revisions;
  • improve logging, recovery, and robustness in the pipeline.

Conclusion

This project successfully developed a software-hardware integration framework for the Galassia-5 satellite payload subsystem. The work covered camera integration, SpaceWire support, Ku-band radio control, AI inference, OS bring-up, and end-to-end timing validation. The final benchmark results show that the image-processing pipeline runs in approximately 4.3 to 4.5 seconds per image, which is well within the 20-second mission requirement. fileciteturn0file0

The project therefore makes a clear contribution: it shows that the Galassia-5 payload pipeline can be integrated, tested, and validated on mission-relevant hardware, while also documenting the technical trade-offs and remaining work needed for final deployment. Hailo was the correct inference choice because it enables practical real-time performance. ZSOM-F01 was the correct platform choice because it is the actual mission payload computer and gives the most meaningful result for final integration.
